# 품질 검수 체크리스트

## 개요

품질 검수 체크리스트는 프로젝트 산출물이 요구사항을 충족하고 릴리즈 준비가 완료되었음을 체계적으로 확인하기 위한 표준 도구입니다. 본 문서는 SK하이닉스 IT 프로젝트의 단계별 품질 검수 기준을 정의합니다.

---

## 1. 산출물 인수 기준 (Acceptance Criteria)

산출물 인수는 아래 조건을 모두 충족하는 경우에만 승인됩니다.

### 1.1 기능 요건 인수 기준

- [ ] 요구사항 명세서(SRS)에 정의된 모든 기능이 구현되었는가?
- [ ] 각 기능의 사용자 수용 테스트(UAT)가 완료되었는가?
- [ ] 기능별 합격 기준(Pass Criteria)이 100% 통과되었는가?
- [ ] 미구현 기능이 있을 경우 PO의 서면 승인을 받았는가?

### 1.2 비기능 요건 인수 기준

| 항목 | 기준 | 확인 여부 |
|------|------|-----------|
| 응답 시간 | 일반 조회 API 2초 이내 | [ ] |
| 처리 성능 | 최대 동시 접속자 처리 목표치 충족 | [ ] |
| 가용성 | 운영 시간 내 99.5% 이상 | [ ] |
| 보안 | OWASP Top 10 취약점 점검 통과 | [ ] |
| 접근성 | 웹 접근성 지침(WCAG 2.1) AA 수준 | [ ] |

---

## 2. 테스트 완료 기준

### 2.1 단위 테스트 (Unit Test)

- [ ] 전체 코드 커버리지 80% 이상 달성
- [ ] 핵심 비즈니스 로직 커버리지 95% 이상 달성
- [ ] 모든 단위 테스트 통과 (실패 0건)
- [ ] 테스트 코드 코드 리뷰 완료

### 2.2 통합 테스트 (Integration Test)

- [ ] 모든 API 엔드포인트 연동 테스트 완료
- [ ] 외부 시스템 연계 테스트 완료 (DB, 메시지 큐, 외부 API)
- [ ] 에러 케이스 및 경계값 테스트 완료
- [ ] 통합 테스트 전체 통과 (실패 0건, 경고 5건 이하)

### 2.3 성능 테스트 (Performance Test)

- [ ] 정상 부하 조건(Normal Load)에서 응답 시간 기준 충족
- [ ] 최대 부하 조건(Peak Load)에서 오류율 1% 미만
- [ ] 스트레스 테스트 후 시스템 정상 복구 확인
- [ ] 메모리 누수(Memory Leak) 없음 확인

### 2.4 보안 테스트 (Security Test)

- [ ] 정적 애플리케이션 보안 테스트(SAST) 완료 및 고위험 취약점 0건
- [ ] 동적 애플리케이션 보안 테스트(DAST) 완료
- [ ] SQL Injection, XSS, CSRF 취약점 점검 완료
- [ ] 인증·인가 로직 검증 완료
- [ ] 민감 데이터 암호화 적용 확인

---

## 3. 배포 전 릴리즈 체크리스트

### 3.1 코드 및 형상 관리

- [ ] 모든 기능 브랜치가 main 브랜치에 병합되었는가?
- [ ] 코드 리뷰가 완료되고 승인을 받았는가?
- [ ] 릴리즈 버전 태그(Tag)가 생성되었는가? (예: v1.2.0)
- [ ] 변경 이력(CHANGELOG)이 업데이트되었는가?

### 3.2 배포 환경 준비

- [ ] 운영 환경 인프라가 배포 요건을 충족하는가?
- [ ] 데이터베이스 마이그레이션 스크립트가 준비되었는가?
- [ ] 마이그레이션 롤백 스크립트가 준비되었는가?
- [ ] 환경 변수 및 설정 파일이 운영 환경에 맞게 준비되었는가?
- [ ] 배포 파이프라인(CI/CD) 테스트가 완료되었는가?

### 3.3 운영 준비

- [ ] 운영 매뉴얼(Runbook)이 작성되었는가?
- [ ] 모니터링 대시보드 및 알림이 설정되었는가?
- [ ] 장애 대응 절차(Incident Response Procedure)가 수립되었는가?
- [ ] 롤백 계획(Rollback Plan)이 문서화되었는가?
- [ ] 운영팀 이관 교육이 완료되었는가?

### 3.4 문서화

- [ ] API 명세서(Swagger/OpenAPI)가 최신 상태인가?
- [ ] 사용자 매뉴얼이 작성되었는가?
- [ ] 아키텍처 다이어그램이 현행 시스템을 반영하는가?
- [ ] 데이터 흐름도(Data Flow Diagram)가 업데이트되었는가?

## 4. 릴리즈 승인 게이트 (Release Approval Gate)

릴리즈는 아래 승인 게이트를 순서대로 통과해야 합니다.

| 게이트 | 승인 주체 | 기준 |
|--------|-----------|------|
| 기술 품질 게이트 | 개발팀장, QA팀장 | 테스트 완료 기준 100% 충족 |
| 보안 게이트 | 보안담당자 | 고위험 취약점 0건 |
| 운영 준비 게이트 | 인프라팀장 | 운영 환경 및 모니터링 준비 완료 |
| 최종 릴리즈 승인 | PM, 사업 책임자 | 전체 게이트 통과 및 사업 승인 |

---

## 5. 결함 분류 기준 및 처리

| 등급 | 정의 | 처리 기한 |
|------|------|-----------|
| Critical | 시스템 불능, 데이터 손실 발생 | 즉시 수정 후 재배포 |
| High | 핵심 기능 사용 불가, 회피 방법 없음 | 24시간 이내 |
| Medium | 기능 저하되나 회피 방법 존재 | 다음 스프린트 |
| Low | 미적 문제, 사용성 불편 | 백로그 등록 후 우선순위 조정 |

- Critical/High 결함이 존재하는 경우 릴리즈가 차단됩니다.
- Medium/Low 결함은 PO와 PM의 합의에 따라 예외 처리 가능합니다.
