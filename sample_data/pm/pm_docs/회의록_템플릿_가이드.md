# 회의록 템플릿 가이드

## 개요

효과적인 회의록은 의사결정의 투명성을 확보하고 후속 실행을 담보하는 핵심 문서입니다. 본 가이드는 SK하이닉스 프로젝트에서 사용하는 회의록 작성 표준과 유형별 템플릿을 제공합니다.

---

## 1. 회의 유형 및 특성

| 회의 유형 | 주기 | 참석 범위 | 주요 목적 |
|-----------|------|-----------|-----------|
| 킥오프 미팅 | 프로젝트 시작 시 1회 | 전체 이해관계자 | 목표 공유, 역할 확정 |
| 주간 현황 보고 | 매주 | PM, 팀장, 주요 멤버 | 진척 확인, 이슈 공유 |
| 스프린트 플래닝 | 스프린트 시작 | 전체 개발팀, PO | 스프린트 목표 및 작업 선택 |
| 설계 검토 (Design Review) | 설계 완료 시 | PM, 개발팀, 아키텍트 | 기술 검토 및 승인 |
| 이해관계자 보고 | 월 1회 이상 | PM, 임원, 고객사 | 사업 현황 및 주요 결정 보고 |
| 리스크 검토 | 월 1회 이상 | PM, 팀장, QA | 리스크 현황 검토 및 대응 |

## 3. 결정 추적 (Decision Tracking)

### 결정 로그 관리 원칙

- 모든 결정 사항은 **결정 번호(DEC-YYYYMMDD-NNN)** 형식으로 고유 번호를 부여합니다.
- 결정의 근거(Rationale), 관련 제약사항, 결정자를 반드시 기록합니다.
- 번복된 결정은 삭제하지 않고 '번복' 상태로 기록하고 번복 사유를 명시합니다.

### 결정 로그 템플릿

| 항목 | 내용 |
|------|------|
| 결정 번호 | DEC-20250315-001 |
| 결정 내용 | API 통신 방식을 REST에서 gRPC로 변경 |
| 결정 근거 | 내부 서비스 간 지연 시간 50ms 이내 요건 충족 필요 |
| 결정자 | 김민준 (아키텍트) |
| 결정 일시 | 2025-03-15 |
| 영향 범위 | 백엔드 개발팀, 인프라팀 |
| 상태 | 확정 |

## 4. 액션 아이템 관리

### 좋은 액션 아이템의 조건 (SMART)

- **S**pecific: 구체적인 산출물 명시 ("검토"가 아닌 "검토 보고서 작성")
- **M**easurable: 완료 여부를 명확히 판단할 수 있어야 합니다.
- **A**chievable: 현실적으로 달성 가능한 목표여야 합니다 (1명의 담당자 지정 권장).
- **R**elevant: 프로젝트 목표와 연관된 의미 있는 작업이어야 합니다.
- **T**ime-bound: 명확한 완료 기한을 설정합니다.

### 액션 아이템 상태

- **미완료**: 아직 시작하지 않은 상태
- **진행중**: 작업 중인 상태
- **완료**: 산출물이 확인된 상태
- **취소**: 더 이상 필요 없어진 상태

---

## 5. 회의록 배포 및 보관

- 회의 종료 후 **24시간 이내** 초안을 참석자에게 배포합니다.
- 이의 제기 기한: 배포 후 48시간
- 이의 없을 경우 확정 처리하고 프로젝트 문서 저장소에 등록합니다.
- 보관 경로: `프로젝트폴더 / 04_회의록 / YYYY-MM / 회의명_YYYYMMDD.md`
- 보관 기간: 프로젝트 종료 후 3년

---

## 6. 효과적인 회의 운영 원칙

- 회의 전: 안건(Agenda)을 48시간 전에 배포합니다.
- 시간 엄수: 시작과 종료 시간을 반드시 지킵니다.
- 퍼실리테이터: 회의마다 진행자를 명확히 지정합니다.
- 기록자: 진행자와 기록자는 분리합니다.
- 의사결정 명확화: "이것으로 결정하겠습니다"라고 명시적으로 선언합니다.
